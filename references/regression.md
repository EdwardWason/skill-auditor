# 回归审计方法论

**When to read**: Phase 5 回归审计模式时。本文件定义回归审计的差异对比规则和报告格式。

---

## 回归审计触发条件

**触发词**："回归审计 <skill>" / "重新审计 <skill> 对比上次"

**前提**：被审计 Skill 目录下存在 `.audit-report.md`（上次审计报告）

**前提不满足时**：降级为首次全量审计，提示"未找到上次审计报告，将执行首次全量审计"

---

## 回归审计流程

```
1. 读取上次报告 <skill-dir>/.audit-report.md
2. 备份上次报告 → <skill-dir>/.audit-report-prev.md
3. 重跑全量审计（8 维度）
4. 生成新报告 → <skill-dir>/.audit-report.md
5. 对比 prev 和 new，生成回归差异报告
```

---

## 差异对比规则

### Finding 状态判定

对上一次报告中的每个 Finding，在新报告中查找对应项：

| 上次状态 | 本次状态 | 判定 | 标记 |
|---------|---------|------|------|
| 🚨/⚠️/💡 FAIL | ✅ PASS 或消失 | **已修复** | ✅ FIXED |
| 🚨/⚠️/💡 FAIL | 同维度同位置仍 FAIL | **未修复** | ❌ UNFIXED |
| 🚨/⚠️/💡 FAIL | 同维度但位置变化 | **部分修复/变化** | ⚠️ CHANGED |
| 无（上次无此问题） | 🚨/⚠️/💡 FAIL | **新增问题** | 🆕 NEW |
| ℹ️ FYI | 仍存在 | **信息保持** | ℹ️ SAME |

### Finding 匹配方法

如何判断"上次 Finding #N"和"本次 Finding #M"是同一个问题：

1. **维度+检查项编号一致**：如 `[T-凭证泄露]` 匹配 `[T-凭证泄露]`
2. **位置一致**：`file:line` 相同或同一文件
3. **问题描述相似**：核心问题一致（位置变化但问题相同 → CHANGED）

如果维度+检查项一致但位置变化 → CHANGED
如果维度+检查项+位置都一致 → UNFIXED
如果维度+检查项一致且本次无该 Finding → FIXED

## 回归审计特殊场景

### 场景1：上次报告格式不标准

如果上次报告不是 skill-auditor 生成的标准格式（如手动写的审计笔记）：
- 尝试提取 Findings 信息
- 无法提取 → 降级为首次全量审计，提示"上次报告格式不标准，无法对比"

### 场景2：Skill 版本号变化

如果上次的 Skill 版本和本次不同（如 v1.0 → v1.1）：
- 正常执行回归审计
- 在报告头部标注"Skill 已从 v1.0 升级到 v1.1"
- 新版本引入的问题标记为 🆕 NEW

### 场景3：上次报告文件被删除

如果 `.audit-report.md` 不存在但 `.audit-report-prev.md` 存在（用户误删）：
- 使用 `.audit-report-prev.md` 作为上次报告
- 提示"使用 .audit-report-prev.md 作为上次报告"

### 场景4：回归审计连续执行

如果短时间内连续执行回归审计（如 1 小时内）：
- 正常执行
- 提示"距离上次审计仅 X 分钟，建议修复后再回归"

---

## 回归审计的限流

为避免无意义回归（未修复就回归）：
- 回归审计前检查 Skill 文件修改时间
- 如果上次审计后 Skill 文件无修改 → 提示"Skill 自上次审计后无修改，回归审计无意义"
- 用户强制要求 → 继续执行
