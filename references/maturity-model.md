# 三级成熟度模型

**When to read**: Phase 0 执行成熟度判定时。本文件定义 L1/L2/L3 三级的识别信号、判定规则和各等级审计范围。

---

## 成熟度等级定义

| 等级 | 名称 | 典型状态 | 用户意图 |
|------|------|---------|---------|
| **L1** | 原型阶段 | 刚创建，功能未完善，可能只有 SKILL.md | 验证基本结构，不求发布 |
| **L2** | 迭代阶段 | 功能基本完成，正在优化迭代 | 持续改进，暂不发布 |
| **L3** | 发布阶段 | 功能稳定，准备发布或已发布迭代 | 发布前全面体检 |

---

## 识别信号矩阵

| 信号 | L1 原型 | L2 迭代 | L3 发布 |
|------|---------|---------|---------|
| references/ 目录 | 不存在 或 0 文件 | 存在且 1-2 文件 | 存在且 ≥3 文件 |
| SKILL.md 行数 | < 50 行 | 50-200 行 | 50-200 行 |
| version 字段 | 0.x.x 或无 | ≥ 1.0.0 | ≥ 1.0.0 且最近 bump |
| CHANGELOG.md | 不存在或空 | 存在且有 ≥1 条 | 存在且最近条目与 version 一致 |
| scripts/ 目录 | 不存在 | 可选 | 存在（如有代码） |
| README.md | 不存在或极简 | 存在 | 存在且中英双语 |
| .claude-plugin/ | 不存在 | 存在 | 存在且 plugin.json 完整 |
| 用户明示 | "刚创建"/"草稿"/"原型" | "在迭代"/"开发中"/"改进中" | "准备发布"/"要上线"/"发布前" |

---

## 判定规则

### 自动识别流程

```
1. 收集 8 个信号的状态
2. 统计每个等级命中的信号数
3. 命中 ≥3 个信号的等级即为该等级
4. 若跨等级信号混杂（如 3个L1 + 3个L2）→ 取较高等级（L2）
5. 若所有等级都 <3 个信号 → 默认 L1（保守判定）
```

### 用户明示优先

**用户明示信号优先级最高**：
- 用户说"准备发布" → 直接判 L3，不论其他信号
- 用户说"在迭代" → 直接判 L2
- 用户说"刚创建/原型" → 直接判 L1

### 判定示例

**示例1**：Skill 有 references/(2文件) + version 1.2.0 + CHANGELOG 有更新 + SKILL.md 180行
→ L2 命中 4 信号 → **判 L2**

**示例2**：Skill 无 references/ + version 0.1.0 + 无 CHANGELOG + SKILL.md 30行
→ L1 命中 4 信号 → **判 L1**

**示例3**：Skill 有 references/(5文件) + version 3.0.0 + CHANGELOG 最新条目 v3.0.0 + README 中英双语 + plugin.json 完整
→ L3 命中 5 信号 → **判 L3**

**示例4**：用户说"准备发布"，但 Skill 实际只有 references/(1文件)
→ 用户明示优先 → **判 L3**（但审计报告会标注"references 文件数不足，建议补充"）

## 各等级审计范围

### L1 原型阶段

**审计范围**（3 项）：
- **S（精简）**：S1-S6（frontmatter + name + description + 4模块 + 行数）
- **A（精简）**：A1-A2（3正例 + 2反例，不跑触发词冲突检测）
- **D1**：SKILL.md 内部引用一致性

**跳过维度**：T / E / C / P / Q（原型阶段不要求安全/有效性/竞争/平台/代码质量）

**严格度**：
- 只报 Critical
- Important 以下标记为"建议"而非"问题"
- 不输出综合评分（维度不足），只输出"基础结构检查通过/不通过"

**典型 Findings**：
- 🚨 description 缺失（Critical）
- 🚨 4 模块缺任一（Critical）
- 💡 建议补充 references/（Minor，标记为"建议"）

### L2 迭代阶段

**审计范围**（6 维度）：S / T / A / E / D / Q

**跳过维度**：C（同类竞争）/ P（平台合规）—— 未到发布阶段，不需要

**严格度**：
- Critical 必修
- Important 建议修（报告中标"建议"，不阻止后续操作）
- Minor 记录但不强制

**典型 Findings**：
- 🚨 凭证泄露（Critical，T 维度）
- ⚠️ SKILL.md 超过 200 行（Important，S 维度）
- ⚠️ 缺少 Do NOT 范围（Important，A 维度）
- 💡 类型注解缺失（Minor，Q 维度）

**确认点3 选项**：A. 再做回归 / C. 结束（**不显示 B 发布选项**）

### L3 发布阶段

**审计范围**（全量 8 维度）：S / T / A / E / C / P / D / Q

**严格度**：
- Critical 必修
- Important 必修（L3 严格度，不修不能发布）
- Minor 建议修

**典型 Findings**：
- 🚨 YARA 触发词（Critical，T/P 维度）
- ⚠️ Missing User Warnings（Important，P 维度）
- ⚠️ 与已有 Skill 重复度高（Important，C 维度）
- 💡 命名规范（Minor，Q 维度）

**确认点3 选项**：A. 再做回归 / B. 发布 / C. 结束（**显示 B 选项**）

**发布前置条件**（确认点4 显示 A 选项的前提）：
- 回归审计综合分 ≥ 7.0
- 无 Critical 未修复
- 无 Important 未修复（L3 要求）
- 成熟度 = L3

## 成熟度覆盖

用户可强制指定等级，覆盖自动识别：

**覆盖方式**：
- "对 wx-peitu 做 L3 全量审计" → 强制 L3
- "这只是原型，做 L1 精简审计就行" → 强制 L1
- "技能审计 wx-peitu，全量审计" → 强制 L3（"全量"= L3）

**覆盖规则**：
- 用户明示覆盖 > 自动识别
- 覆盖后在确认点1展示"用户指定 L3（自动识别为 L2）"
- 用户仍可在确认点1再次调整

---

## 成熟度与整改模式

| 等级 | 整改模式 | 整改范围 |
|------|---------|---------|
| L1 | 支持 | 只修 Critical |
| L2 | 支持 | 修 Critical + Important（建议） |
| L3 | 支持 | 修 Critical + Important（必须）+ Minor（建议） |

**整改后回归**：
- L1 整改后 → 回归只检查 L1 范围
- L2 整改后 → 回归检查 L2 范围
- L3 整改后 → 回归检查全量 8 维度

---

## 成熟度与发布

| 等级 | 确认点3 发布选项 | 确认点4 |
|------|----------------|---------|
| L1 | ❌ 不显示 | ❌ 不触发 |
| L2 | ❌ 不显示 | ❌ 不触发 |
| L3 | ✅ 显示 | ✅ 触发（需用户明确授权） |

**L1/L2 想发布怎么办**：
- 用户在确认点3 看不到发布选项
- 如用户坚持发布 → 提示"当前为 L<L>阶段，建议先升级到 L3（补全 references/CHANGELOG 等）再做发布审计"
- 用户可强制覆盖为 L3 → 重新审计 → 通过后可发布

## 成熟度识别的特殊场景

### 场景1：Skill 刚创建完

- 信号：references/ 可能 0-1 文件 + version 1.0.0 + CHANGELOG 可能有 1 条
- 判定：L1 或 L2（取决于 references 数量）
- 建议：如果是刚创建，默认 L1，用户确认后可升级到 L2

### 场景2：Skill 已发布但有新版本迭代

- 信号：references ≥3 + version 最近 bump + CHANGELOG 最新条目与新 version 一致
- 判定：L3
- 建议：L3 全量审计，确保迭代后仍符合发布标准

### 场景3：Skill 长期未维护

- 信号：CHANGELOG 最近条目时间 > 6个月 + version 未变
- 判定：L2（已发布但不在活跃迭代）
- 建议：L2 审计，检查是否有过期依赖或失效 API

### 场景4：用户不确定等级

- 用户说"我不确定这是什么阶段"
- 判定：自动识别 + 在确认点1详细展示各信号状态
- 让用户基于信号自行决定
